# 2026-03-23 EMR-Style Memory for Baseline3 Leaf Loop

## 목적
- Tree 없이 동작하는 `baseline3_leaf_only_loop`에 EMR-style memory를 붙여, iterative query refinement에서 memory 자체의 효과를 분리해서 보려는 목적.
- Retrieval / ranking / scoring은 그대로 두고, rewrite prompt에만 memory를 추가하는 prompt-only ablation으로 구현.
- 현재 paper-faithful main setting은 `accumulated`다.

## 구현 핵심
- 대상 실행 경로
  - `src/run_leaf_rank.py`
  - `src/bash/baselines/run_baseline3_leaf_only_loop_emr.sh`
- 원본 launcher인 `src/bash/baselines/run_baseline3_leaf_only_loop.sh`는 수정하지 않고 유지.
- 새 인자
  - `--leaf_emr_memory_mode off|accumulated`
  - `--leaf_emr_history_rank_topk`
  - `--leaf_emr_doc_topk`
  - `--leaf_emr_sent_topk`
  - `--leaf_emr_compression on|off`
  - `--leaf_emr_memory_max_tokens`
- 새 prompt
  - `agent_executor_v1_icl2_emr_memory`
  - 기존 `agent_executor_v1_icl2`를 유지하고, memory-aware variant를 분리함.

## Paper-Faithful 기준
- `History of Recent Actions`는 누적된다.
- `Memory of Documents`도 누적 document pool을 기반으로 만든다.
- document compression은 per-document top-k가 아니라, 누적 pool 전체 문장에 대해 query-conditioned global filtering을 수행한다.
- 현재 구현은 논문 문구에 맞춰 global top-k sentence selection을 사용하며, 기본값은 `k=10`이다.

## 기본값의 출처
- `--leaf_emr_sent_topk=10`
  - EMR 논문 본문의 global filtering 설명에 맞춘 값이다.
  - 의미는 "문서별 sentence top-k"가 아니라, cumulative document pool 전체에서 globally selected sentence top-k이다.
- `--leaf_emr_compression=on`
  - paper-faithful 기본값이다.
  - `on`이면 cumulative document pool 위에서 global sentence filtering을 적용한다.
  - `off`는 ablation 용도로, full document text를 그대로 memory에 넣고 token budget만 적용한다.
- `--leaf_emr_memory_max_tokens=0`
  - `0`은 auto 모드다.
  - rewrite prompt 전체를 실제 LLM tokenizer로 토크나이즈하고, model context window 안에 들어오도록 document memory를 자동 축소한다.
  - 양수 값을 직접 주면 document memory block token 수의 상한으로 동작한다.

## Mode 해석
- `accumulated`
  - paper-faithful main setting.
  - 이전 iteration들에서 본 문서를 deduplicated bank로 유지한다.
  - `leaf_emr_compression=on`이면 rewrite 시에는 이 누적 pool 전체 문장들을 query-conditioned로 global scoring하고, top-k sentences만 남긴다.
  - top-k에 기여한 문서만 `Memory of Documents`에 남으므로 irrelevant documents는 문장 단계에서 암묵적으로 prune된다.
  - `leaf_emr_compression=off`이면 누적 pool의 full document text를 그대로 memory에 넣고, token budget만 적용한다.

## Baseline 실행
- 메모리 없는 기존 baseline은 그대로 `src/bash/baselines/run_baseline3_leaf_only_loop.sh`로 실행한다.
- 같은 EMR launcher 표면에서 memory만 끄고 싶으면 `LEAF_EMR_MEMORY_MODE=off bash src/bash/baselines/run_baseline3_leaf_only_loop_emr.sh`를 사용하면 된다.

## Memory 구조
### 1. History of Recent Actions
- EMR 스타일을 따라 plain text log로 저장.
- 각 항목은 다음 정보를 포함.
  - `Query`
  - `Ranks` = iteration별 top retrieved `doc_id` 목록
- 현재 step은 rewrite가 끝난 뒤에 append되므로, 다음 prompt부터 보이게 구현.

### 2. Memory of Documents
- 누적 document pool에서 시작한다.
- 각 문서를 sentence split한 뒤, 모든 문장을 하나의 글로벌 후보 집합으로 합친다.
- MiniLM cross-encoder (`cross-encoder/ms-marco-MiniLM-L6-v2`)로 query-conditioned sentence scoring을 수행한다.
- top-k overall sentences만 남긴다.
- 선택된 sentence를 가진 문서들만 `Memory of Documents`에 나타난다.
- sentence split은 `spacy`가 있으면 sentencizer를 사용하고, 없으면 regex fallback 사용.

## 주의점
- 이번 구현은 prompt-only memory 실험이다. retrieval metric, retrieval order, evaluation 방식은 바꾸지 않았다.
- 논문 재현 관점에서는 `accumulated + compression=on + global sentence filtering(top-k=10)`이 중심이다.

## 2026-03-23 추가: Round6 Expandable + EMR Memory Overlay
- 대상 실행 경로는 기존 `src/bash/round5/run_round6_expandable.sh`를 수정하는 대신, 복사본 launcher인 `src/bash/round5/run_round6_expandable_emr.sh`를 새로 추가했다.
- 목적은 `round6`의 tree/controller memory는 유지한 채, rewrite prompt에만 EMR-style long-term memory를 얹어서 end nDCG stability를 개선할 수 있는지 보는 것이다.
- 즉 `archive_records`, `fusion_bank_*`, ended-reseat branch update 로직은 그대로 두고, rewrite prompt에만 `History of Recent Actions`와 `Memory of Documents`를 주입한다.

### Round6 기본값 정렬
- `round6` EMR 기본값은 `src/bash/baselines/run_baseline3_leaf_only_loop_emr.sh`와 맞췄다.
  - `topk=10`
  - `sent_topk=10`
  - `compression=on`
  - `memory_max_tokens=0`
- 기본값은 `src/hyperparams.py`의 공용 EMR 상수를 사용하도록 맞춰서, leaf-only EMR과 round6 EMR이 같은 기본값을 보게 했다.

### History Formatter 변경
- baseline3와 round6 모두 같은 formatter를 사용하도록 공용 helper로 분리했다.
- history format은 이전의 `Action : REFINE`를 제거하고 아래처럼 단순화했다.
  - `[idx] Query : <query> Ranks : <doc ids>`
- 이유는 현재 iterative rewrite setting에서는 별도의 LLM action selection이 없고, 실제로 필요한 정보는 query trajectory와 retrieved doc ids이기 때문이다.

### Round6에서 Memory Source를 이렇게 잡은 이유
- `round6` EMR memory의 document source는 `pre_hits`가 아니라 이전 iteration들의 `eval_hits`다.
- 이유는 현재 `ours`의 부족한 점이 peak 성능보다 end 성능 유지에 더 가깝다고 봤기 때문이다.
- 그래서 현재 live rewrite context(`leaf_descs`)와 별개로, 과거 iteration에서 실제 post-rewrite retrieval로 얻었던 문서를 장기 memory로 보존하도록 했다.
- 현재 iteration의 `eval_hits[:topk]`는 iteration이 끝난 뒤 bank에 append되고, 같은 `topk` 값으로 history의 `Ranks` 길이도 결정된다.

### pre_hit overlap을 유지한 이유
- round6 EMR memory에서는 현재 `pre_hit_doc_ids`와 겹치는 archived doc을 일부러 제거하지 않았다.
- 이유는 중복 제거를 추가 규칙으로 넣으면, round6 EMR이 baseline3 EMR의 단순 누적-memory semantics에서 멀어지기 때문이다.
- 현재 구현은 `leaf_descs`가 live context, `Memory of Documents`가 long-term memory라는 역할 분리는 유지하되, overlap 자체는 허용한다.

### leaf reached + reseat 시점
- leaf node에 도달해서 ended-reseat가 일어나더라도, 그 iteration의 `query_post`와 `eval_doc_ids`는 정상적으로 EMR history/document bank에 append된다.
- 다만 same-iteration에 바로 prompt를 다시 만드는 구조는 아니므로, reseat iteration에서 생성된 EMR memory는 다음 iteration rewrite prompt에서 보인다.
- 이 타이밍은 current-run 정보가 same-iteration prompt에 leak되지 않도록 하기 위한 의도다.

### Prompt Example
- 아래는 baseline3 EMR과 round6 EMR이 공통으로 사용하는 memory-aware rewrite prompt의 핵심 부분 예시다.
- 실제 템플릿은 `src/rewrite_prompts.py`의 `agent_executor_v1_icl2_emr_memory`를 사용한다.

```text
Original Query:
How best to count bees entering and leaving a hive to measure hive activity?

Previous Rewritten Query:
Poisson process style counting model for hive traffic, arrival rate estimation, event count variance.

History of Recent Actions:
[1] Query : How best to count bees entering and leaving a hive to measure hive activity? Poisson process style counting model for hive traffic, arrival rate estimation, event count variance. Ranks : Poisson_distribution, Count_data, Arrival_process
[2] Query : How best to count bees entering and leaving a hive to measure hive activity? Event-count stochastic model for repeated independent arrivals, insect traffic monitoring. Ranks : Event_count_models, Time_series_counting, Queueing_arrivals

Memory of Documents:
[Poisson_distribution] Poisson distribution models the number of independent events in a fixed interval ... widely used for arrival-count processes ...
[Arrival_process] Arrival processes describe event arrivals over time ... count observations can often be modeled with Poisson assumptions ...
[Count_data] Count data analysis covers event-frequency modeling, rate estimation, and variance assumptions for repeated observations ...

Context Summaries:
Monitor the hive entrance and record incoming/outgoing bees over fixed intervals.
Compare temporal count patterns to estimate colony activity level.
```

- 여기서 `History of Recent Actions`는 query trajectory와 top retrieved `doc_id` trace를 보여준다.
- `Memory of Documents`는 누적 bank에서 가져온 document snippet이다.
  - `compression=on`이면 global sentence filtering 이후의 snippet이 들어간다.
  - `compression=off`이면 full document text가 token budget 안에서 들어간다.
- `Context Summaries`는 현재 iteration에서 방금 본 live evidence이고, `Memory of Documents`는 이전 iteration들에서 보존된 long-term evidence다.
