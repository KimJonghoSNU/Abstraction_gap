  ## 알고리즘 상세(결정 완료)

  각 sample별 iter 처리:

  1. 현재 branch 상태에서 local retrieval (pre-rewrite)
      - iter0에서 selected_branches가 비어 있으면 root 전체 하위 leaf를 pool로 사용
      - leaf pool에서 query(last_query 없으면 original query)로 top100 retrieval
      - 이 결과를 rewrite context(leaf_descs)로 사용
  2. rewrite (agent_executor_v1)
      - prompt 입력: original_query, previous_rewrite, leaf_descs, branch_descs(비워도 됨)
      - 출력 JSON의 Possible_Answer_Docs를 파싱하고 concat하여 new_query 생성
      - 캐시 hit/miss 모두 동일한 state 포맷으로 기록
  3. local retrieval (post-rewrite, branch selection용)
      - 동일 leaf pool에서 new_query로 top100 retrieval 수행
      - 이 랭킹을 기준으로 child 후보 branch MRR 계산
  4. 다음 branch 선택 (global top-B)
      - 후보 집합: selected_branches의 direct children 전체
      - 후보 branch 점수: best reciprocal rank
      - 전 후보를 합쳐 정렬 후 B=max_beam_size개 선택
      - iter0인 경우 root child 후보에서 동일 방식으로 초기 beam 생성
  5. 평가/로그
      - main ranking은 post-rewrite local retrieval 결과 leaf->doc_id 순서
      - nDCG@10, Recall@10/100/all, Coverage
      - iter_records에 아래를 저장:
          - selected_branches_before/after
          - candidate_branch_scores (MRR + tie-break score)
          - query_pre, query_post, rewrite_docs
          - local_paths/doc_ids, metrics

## 2026-03-04 업데이트: Selector mode 확장 (`retriever_slate` / `maxscore_global` / `meanscore_global` / `max_hit_global`)

### 구현 반영
- `src/hyperparams.py`
    - `--round5_selector_mode {retriever_slate,maxscore_global,meanscore_global,max_hit_global}` 추가.
    - default는 `retriever_slate`.
    - `round5_mrr_pool_k`는 selector local top-K(후보 점수 계산용) 의미로 유지.
- `src/run_round5.py`
    - `sample.update(...)`로 baseline traversal 상태 업데이트는 그대로 유지.
    - 단, selector mode가 `maxscore_global`/`meanscore_global`/`max_hit_global`일 때만 beam 선택 override:
        1) 현재 selected branch들의 direct child branch를 후보로 수집
        2) 현재 selected branch descendant leaf pool에서 top-K retrieval
        3) 후보 branch별 점수 계산:
            - `maxscore_global`: 매칭 leaf 점수 중 최대값
            - `meanscore_global`: 매칭 leaf 점수 평균값
            - `max_hit_global`: 매칭 leaf hit 개수(상위 K 내 포함 수)
        4) global top-B 선택 후 expandable state path에 매핑해 `beam_state_paths` 교체
        5) 매핑 실패/후보 없음이면 같은 iter의 retriever-slate 결과 유지(fallback)
    - `iter_records`에 selector 진단 필드 저장:
        - `selector_mode`, `selector_pick_reason`
        - `selector_candidate_branch_count`
        - `selector_scored_top` (상위 일부 score row)
- `src/bash/round5/run_round5.sh`
    - selector를 단일 고정이 아니라 loop로 실행:
        - 기본: `retriever_slate`, `maxscore_global`, `meanscore_global`, `max_hit_global`
    - `ROUND5_SELECTOR_MODES` env로 실행 조합 제어 가능.

### 실행 예시
```bash
# 기본 4개 모드(retriever_slate + maxscore_global + meanscore_global + max_hit_global) 순회
bash src/bash/round5/run_round5.sh

# 선택 모드 지정 실행
ROUND5_SELECTOR_MODES="retriever_slate maxscore_global meanscore_global max_hit_global" \
bash src/bash/round5/run_round5.sh
```

### 해석 포인트
- 현재 설계는 "파이프라인 동일 + branch score 계산식만 변경" 실험이다.
- 따라서 비교 시 핵심은 selector mode별:
    - `BranchHit@B`, `BranchPrecision@B`
    - `nDCG@10` 변화
    - `selector_pick_reason` 분포(얼마나 실제 override가 적용됐는지)

## 2026-03-03 업데이트: Legacy 알고리즘(현재 실행 기준)

현재 `results/BRIGHT/ndcg_summaryround5.csv`는 **legacy round5** 결과로 해석한다.

주의: 이름에 `mrr_selector`가 남아 있어도, 현재 코드의 실제 분기 선택은 `MRR global top-B`가 아니라 **retriever-slate + beam update** 방식이다.

### 용어 정리

- `cumulative reached leaf pool`: 이전 iteration까지 실제로 도달한 leaf들의 합집합.
- `selected branches`: 현재 beam에 살아있는 branch 경로들.
- `slate`: 각 branch state에서 다음으로 점수화할 후보 노드 목록.
- `selector local leaf pool`: `maxscore_global`/`meanscore_global`에서만 쓰는 국소 pool. 현재 `selected branches` 하위 leaf들의 합집합(비면 전체 leaf fallback).

### Iteration t 알고리즘 (코드 기준)

1. `query_pre` 결정
    - `sample.query`가 있으면 사용, 없으면 `original_query` 사용.
    - 따라서 t>0에서는 이전 iteration의 `query_post`가 자동 재사용된다.

2. pre-rewrite retrieval (rewrite 컨텍스트 수집)
    - 검색 대상 pool은 `cumulative reached leaf pool`.
    - pool이 비어 있으면 전체 leaf를 사용.
    - `query_pre`로 top-`round5_mrr_pool_k` retrieval 수행.
    - 상위 hit의 desc를 `leaf_descs`로 만들어 rewrite 입력에 사용.
    - 중요한 점: 이 pool은 branch별로 줄어드는 구조가 아니라, iteration이 진행될수록 누적되는 구조다.

3. rewrite (`agent_executor_v1`)
    - (legacy 기준) 입력: `original_query`, `previous_rewrite`, `leaf_descs`.
    - `branch_descs`는 legacy rewrite 경로에서 비워서 전달된다.
    - 출력: `Possible_Answer_Docs`를 파싱하고 `rewrite` 문자열 생성.
    - `query_post = original_query + rewrite` (rewrite가 비면 `query_pre` 유지).

4. post-rewrite retrieval (평가용)
    - 같은 pool(`cumulative reached leaf pool`, 비면 전체 leaf)에서 `query_post`로 retrieval.
    - top-k는 `flat_topk`를 사용한다.
    - 이 결과로 nDCG/Recall/Coverage 계산.

5. branch 선택 (selector mode별)
    - 각 sample의 현재 beam state마다 `slate`를 생성한다.
    - 기본 slate는 현재 state의 child들로 만들고, leaf-cluster 조건에서는 predicted leaf 기반 항목이 추가될 수 있다.
    - `retriever_slate` 모드:
        - 각 slate 후보 노드 점수:
            - `score = dot(query_emb, node_emb)` (임베딩 정규화 상태라 cosine과 동일)
            - `score_01 = clip((score + 1) / 2, 0, 1)`
        - `score_01` 내림차순 ranking을 `sample.update(...)`에 전달.
        - `update` 내부에서 `path_relevance` 기준 top-`max_beam_size`를 다음 selected branches로 선택.
    - `maxscore_global` / `meanscore_global` / `max_hit_global` 모드:
        - 먼저 `sample.update(...)`를 동일하게 수행해 baseline beam 후보를 만든다.
        - 그 다음 override 후보를 계산:
            1) 후보 branch = 현재 `selected_branches_before`의 direct child branch들
            2) 검색 pool = 해당 branch들의 하위 leaf 합집합(`selector local leaf pool`)
            3) `query_post`로 top-`round5_mrr_pool_k` retrieval
            4) 후보 branch 점수:
                - `maxscore_global`: 매칭 leaf score의 최대값
                - `meanscore_global`: 매칭 leaf score의 평균값
                - `max_hit_global`: 매칭 leaf hit 개수(상위 K 내 포함 수)
            5) global top-B branch를 고른 뒤 expandable path로 매핑해 beam override
        - 매핑 실패/후보 없음/매칭 없음이면 retriever-slate 결과를 그대로 유지(fallback).

    - 정리: 질문한 문장처럼 "확장하겠다고 결정한 branch node 하위의 leaf node들"을 쓰는 건 **selector 점수 계산 단계에서만 맞다**.
      rewrite/eval retrieval pool은 여전히 `cumulative reached leaf pool`이다.

6. 상태 업데이트
    - iteration 종료 시점의 selected branches, retrieval 결과, metrics를 `iter_records`에 저장.
    - 새로 도달한 leaf는 다음 iteration의 `cumulative reached leaf pool`에 추가.

핵심: LLM은 query rewrite만 담당하고, branch 확장은 retriever 점수 기반 slate/beam 메커니즘으로 진행된다.

## 2026-03-04 업데이트: Accumulation의 "새 leaf gold 발굴" 분석

분석 스크립트: `scripts/analyze_round5_accum_new_gold.py`

실행 커맨드:

```bash
python scripts/analyze_round5_accum_new_gold.py \
    --glob_pattern 'results/BRIGHT/*/round5/**/all_eval_sample_dicts.pkl' \
    --include 'S=round5_mrr_selector_accum' 'RPN=agent_executor_v1' 'MaxBS=5' 'NumI=10'
```

설정:
- `topk=10`
- `min_iter=1` (iter0 제외)
- 정의:
    - `new_top10`: 현재 iter `local_doc_ids[:10]` 중, 이전 iter들에서 한 번도 안 나온 문서
    - `P(new_gold@10)`: `new_top10` 안에 gold가 1개 이상 있을 확률
    - `P(new_only_gold@10)`: `old_top10`에는 gold가 없고 `new_top10`에만 gold가 있는 확률

모델별 평균(5 subsets):
- `Qwen3-30B-A3B-Instruct-2507`
    - `P(gold@10)=0.6334`
    - `P(new_gold@10)=0.0012`
    - `P(new_only_gold@10)=0.0000`
    - `avg_new_slots@10=0.0714`
    - `P(has_new_slot)=0.0276`
    - `P(new_gold@10 | has_new_slot)=0.0393`
- `Qwen3-4B-Instruct-2507` 계열
    - `P(gold@10)=0.6286`
    - `P(new_gold@10)=0.0006`
    - `P(new_only_gold@10)=0.0000`
    - `avg_new_slots@10=0.0596`
    - `P(has_new_slot)=0.0242`
    - `P(new_gold@10 | has_new_slot)=0.0239`

해석:
- accumulation 이후 iter에서 `top-10`은 대부분 기존(이미 본) 문서로 구성된다.
- `new_top10` 자체가 매우 드물고(`P(has_new_slot)≈2~3%`), 그 새 문서가 gold일 확률도 매우 낮다.
- 따라서 현재 round5 accumulation 성능 이득은, "새 gold 발굴"보다는 기존 후보 재정렬/안정화 효과에 더 가깝다.

### 추가 확인: \"branch가 틀어지기 전(초반 iter)\"만 보면?

동일 세팅(4B, MaxBS=2, 5 subsets, `round5_mrr_selector` vs `round5_mrr_selector_accum`)에서
iteration window를 잘라서 비교했다.

- `iter <= 3` (초반 구간 평균 nDCG@10):
    - accum: `42.22`
    - non-accum: `42.43`
    - delta: `-0.21` (사실상 큰 차이 없음)

- `iter <= 4` (iter4 포함):
    - accum: `42.29`
    - non-accum: `38.87`
    - delta: `+3.42`

iter별 평균 nDCG@10(5 subsets 평균):
- iter0: accum `41.73` vs non `41.81` (`-0.08`)
- iter1: accum `42.38` vs non `42.28` (`+0.10`)
- iter2: accum `42.47` vs non `42.28` (`+0.19`)
- iter3: accum `42.32` vs non `43.34` (`-1.03`)
- iter4: accum `42.54` vs non `24.62` (`+17.93`)

해석:
- 사용자 가설대로, \"branch가 크게 틀어지기 전\" 구간에서는 accumulation 이득이 거의 없다.
- 실제 이득은 주로 **iter4 이후 non-accum의 급락을 막는 안정화 효과**에서 발생한다.
- 따라서 accum의 본질은 exploration으로 새 gold를 많이 발굴하는 전략이 아니라, 한 번 확보한 후보를 유지하는 memory 효과에 가깝다.

## Query-level 추가 분석: key drift vs key count
