# 2026-03-19 MCTS vs `run_round6_expandable.sh`

이 문서는 `src/bash/round5/run_round6_expandable.sh`가 현재 어떻게 동작하는지와, 새 `src/run_mcts.py`가 무엇을 바꾸는지를 짧게 정리한 메모다.

## 핵심 요약

- `run_round6_expandable.sh`는 현재 `go-explore inspired`라고 부르고 있지만, 실제 controller는 `ended_reseat` heuristic이다.
- 새 `run_mcts.py`는 같은 rewrite / retrieval / metric stack을 유지하고, **branch controller만** `single-path local UCT`로 바꾼다.
- 따라서 이 비교의 요점은 `rewrite policy 비교`가 아니라 `tree controller 비교`다.

## `run_round6_expandable.sh`는 실제로 무엇을 하나

실행 대상은 `src/run_round6.py --round6_expandable_mode ended_reseat`다.

현재 controller는 다음과 같다.

1. cumulative reached leaf pool에서 rewrite evidence retrieval을 한다.
2. rewrite를 만든다.
3. rewritten query로 eval retrieval을 한다.
4. 현재 selected branches의 direct child branch를 retrieval score로 점수화한다.
5. locally ended branch slot이 있으면, leftover expandable frontier를 다시 retrieval score로 채워 넣는다.

즉 이 방식은:

- explicit archive return 없음
- explicit visit count 없음
- explicit value backup 없음
- beam slot reseat heuristic 있음

그래서 이건 엄밀한 MCTS가 아니라, retrieval-scored local search + ended-slot reseat에 가깝다.

## `run_mcts.py`가 바꾸는 것

`run_mcts.py`는 아래는 그대로 유지한다.

- same dataset / tree / node embeddings
- same retriever
- same rewrite prompt and rewrite cache
- same cumulative rewrite/eval retrieval pattern
- same end metrics and `all_eval_sample_dicts.pkl` saving path style

바뀌는 것은 controller뿐이다.

### State

- state = 현재 active branch endpoint 하나
- 구현상 `sample.mcts_active_path`로 저장한다.

### Action

- action = 현재 state의 direct non-leaf child branch 중 하나 선택

### Reward

- reward = 선택된 branch 아래 descendant leaf retrieval score proxy
- 현재 구현은 `--mcts_reward_mode`로 정하고, 기본은 `mean_score`
- retrieval은 `--mcts_rollout_topk` top-k leaf hits를 사용한다.

### Tree policy

- current active state를 local root로 놓고 UCT simulation을 여러 번 수행한다.
- child stats에는 `visits`, `value_sum`, `immediate_reward`를 유지한다.

### Backup

- simulation path를 따라 `visits += 1`, `value_sum += reward`

### Terminal handling

- beam reseat 대신, active branch가 terminal이면 가장 가까운 expandable ancestor로 backtrack한다.
- 이건 현재 코드베이스에서 beam-free single-path search를 안전하게 유지하기 위한 선택이다.

## round6 expandable과의 가장 중요한 차이

### round6 expandable

- 여러 branch를 beam으로 같이 들고 간다.
- ended slot만 leftover expandable path로 다시 앉힌다.
- selection 근거는 매 iter retrieval-scored heuristic row다.
- history는 `prediction_tree + beam state`로 간접적으로만 남는다.

### run_mcts.py

- 한 번에 active branch 하나만 commit한다.
- ended slot reseat는 없다.
- selection 근거가 `visit count + backed-up value`로 명시적이다.
- search history가 `path -> stats` 형태로 남는다.

요약하면:

- round6 expandable: heuristic beam reseat controller
- run_mcts.py: explicit single-path tree-search controller

## 현재 구현에서 의도적으로 제한한 것

- full root-wide MCTS가 아니라 **local MCTS**다.
  - 현재 active branch를 local root로 삼는다.
  - 이유: 기존 prediction tree / rewrite loop를 유지하면서 controller만 바꾸기 위해서다.
- rollout은 one-step retriever reward만 쓴다.
  - multi-step heuristic rollout이나 LLM rollout은 넣지 않았다.
  - 이유: 비용이 크고 비교 실험에서 confound가 커지기 때문이다.
- beam frontier set을 state로 쓰지 않는다.
  - beam-set MCTS는 action space가 너무 커지고, 현재 코드베이스에선 구현 복잡도가 급격히 올라간다.

## 현재 해석 가이드

이 구현으로 보면 다음 질문이 가능해진다.

1. heuristic ended-reseat보다 UCT controller가 gold branch contact를 더 빨리 만드는가?
2. branch reseat heuristic 없이도 terminal branch backtrack만으로 충분한가?
3. query drift가 있을 때, beam reseat보다 visit-backed single-path controller가 더 안정적인가?

반대로 아직 말하면 안 되는 것은 이것이다.

- "이 구현은 정통 full MCTS다"
- "go-explore와 MCTS를 완전히 공정하게 구현했다"

현재 구현은 **controller-only MCTS ablation**으로 보는 것이 맞다.

## How to Claim the Difference

이 비교에서 가장 중요한 원칙은, `go-explore style > MCTS`를 일반론으로 밀지 않는 것이다.

현재 방어 가능한 비교 대상은 이것이다.

- `src/bash/round5/run_round6_expandable.sh`
  - `ended_reseat` controller
  - multi-frontier state를 유지하다가 locally ended slot을 leftover expandable branch로 복구한다.
- `src/bash/round5/run_mcts.sh`
  - `src/run_mcts.py`
  - local UCT controller
  - active branch 하나를 기준으로 visit/value backup을 수행한다.

즉 headline은 `Go-Explore vs MCTS in general`이 아니라, 아래처럼 **task-specific / implementation-specific**이어야 한다.

- 추천 headline:
  - `In this rewrite-driven retrieval setting, recoverable frontier search is a better controller than the current local MCTS-style visit-count search.`
- 더 보수적인 논문용 문장:
  - `Under matched retriever, rewrite, and tree artifacts, the ended-reseat controller outperforms the current local MCTS controller on end/max retrieval quality. This suggests that recoverable branch persistence is more effective than local visit-count search for this information-retrieval task.`

핵심 차이는 이렇게 설명하는 게 가장 정확하다.

- `ended_reseat`
  - alternate branch state를 동시에 유지한다.
  - locally ended branch가 생겨도, 이전에 살아 있던 expandable frontier를 다시 붙일 수 있다.
  - 즉 reward가 바로 안 보이는 branch도 later iteration까지 보존할 수 있다.
- `local MCTS`
  - 현재 active branch 주변에서 repeated one-step value estimation을 수행한다.
  - 선택 근거는 explicit하지만, recoverable frontier를 별도로 보존하지는 않는다.
  - rewrite-driven retrieval처럼 reward target이 iteration마다 바뀌는 setting에서는, local visit count가 delayed branch utility를 놓칠 수 있다.

## What You Should Not Claim

아래 문장은 현재 실험 설계로는 과하다.

- `Go-Explore is better than MCTS in general.`
- `This proves Go-Explore is the right search algorithm for reasoning-intensive retrieval.`
- `This result transfers to full MCTS.`
- `This is a fair comparison against canonical MCTS.`

이유는 분명하다.

- 현재 `run_mcts.py`는 full root-wide MCTS가 아니다.
- rollout reward도 one-step retriever proxy다.
- 즉 비교 대상은 **current local MCTS baseline**이지, MCTS 전체 계열이 아니다.

## Evidence Needed Before You Say "Better"

`better than MCTS`라는 표현은 아래 조건을 만족한 다음에만 써야 한다.

1. 같은 subset set을 사용한다.
2. 같은 retriever와 node embedding artifact를 사용한다.
3. 같은 rewrite prompt를 사용한다.
4. 같은 `num_iters`를 사용한다.
5. 같은 `max_beam_size=1`을 사용한다.
6. 최소 아래를 같이 보고한다.
   - overall mean `end nDCG@10`
   - overall mean `max nDCG@10`
   - per-subset `end nDCG@10`
   - per-subset `max nDCG@10`

숫자가 섞여 있으면 claim도 낮춰야 한다.

- clear win이면:
  - `recoverable frontier search outperforms the current local MCTS controller`
- mixed result면:
  - `the two controllers behave differently, with ended-reseat favoring recoverable branch persistence and local MCTS favoring visit-backed local commitment`

## Mechanistic Reading

만약 `ended_reseat > local MCTS`가 나오면, 가장 설득력 있는 mechanistic reading은 이것이다.

- 이 task의 reward는 stationary하지 않다.
  - rewrite가 query를 바꾸고
  - retrieval pool도 branch state에 따라 바뀐다.
- 그래서 controller가 해야 할 일은 단순한 local value maximization이 아니라, **later payoff가 있을 수 있는 branch state를 잃지 않는 것**이다.
- `ended_reseat`는 그 점에서 recoverable frontier를 직접 모델링한다.
- 반면 현재 `local MCTS`는 visit-backed search를 하더라도, active branch 중심 one-step reward 구조 때문에 delayed activation branch를 보존하는 데 불리할 수 있다.

이 해석은 현재 ended-reseat 분석과도 맞는다.

- partial reseat는 `t+1` immediate gain이 아니라
- `t+2`, `t+3` delayed activation에서 payoff가 나타난다.

즉 이 비교에서 강조할 차이는 `more intelligent planning`이 아니라, **recoverability of frontier states**다.
