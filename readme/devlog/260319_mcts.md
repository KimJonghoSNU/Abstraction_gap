# 2026-03-19 Go-Explore-Inspired Controller vs MCTS Ablation

## Question

- 같은 `frontiercum_qstate` 환경에서
- archived frontier를 recoverable하게 유지하는 controller가
- 같은 retrieval / rewrite / EMR / pool semantics 위에서
- MCTS-style ended-slot controller보다 더 좋은가

즉 비교의 핵심은 **controller**다.
retriever, rewrite prompt, query-state carry, EMR, retrieval pool이 바뀌면 이 비교는 흐려진다.

## Fair comparison target

이 비교에서 `ours`는 아래로 고정한다.

- `frontiercum_qstate`
- `ended_reseat`
- `leftover_expandable`
- retrieval pool = `frontier descendants + cumulative reached leaves`
- EMR on/off도 비교 대상과 같게 맞춤

이걸 `go-explore inspired`라고 부르는 이유는 간단하다.

- promising frontier state를 explicit하게 유지한다
- locally ended branch가 생기면 그 archived frontier로 다시 돌아갈 수 있다
같은 환경 위에서 **ended-slot replacement controller만** 바꾼 버전이다.

유지:

- same retriever
- same node embeddings
- same rewrite prompt
- same `frontiercum_qstate` query-state carry
- same EMR memory behavior
- same retrieval pool semantics
- same eval metrics

변경:

- ended slot이 생겼을 때 어떤 branch로 replacement할지 고르는 규칙만 MCTS-style로 교체

## What still differs conceptually

이 parity 비교에서는 **둘 다 explicit tree-state substrate를 공유하고, 둘 다 current-query로 branch를 다시 평가한다**.
즉 차이는 `state를 저장하느냐` 자체가 아니다.

- ours
    - archived branch 후보를 현재 iteration에서 **바로 reseat score로 고른다**
- MCTS ablation
    - 같은 ended-slot 후보 집합 위에서 **subtree simulation을 더 한 뒤** replacement를 고른다

그래서 이 비교의 핵심은

- `state storage vs no storage`

가 아니라

- `direct one-shot archived-branch reseat`
vs
- `within-event local subtree planning`

이다.

## Biggest difference from the old `run_mcts.py`

old `run_mcts.py`는:

- single active path만 유지
- local UCT를 current root에서 반복
- `frontiercum_qstate` 없음
- EMR parity 없음
- ended-slot reseat 문제가 아니라 single-path control 문제를 풂

새 parity MCTS는:

- `run_round6.py`의 multi-beam / query-state / EMR / pool loop를 그대로 유지
- active-slot continuation은 그대로 둠
- **ended slot replacement만** MCTS로 바꿈

즉 문제 정의가 다르다.

- old: full local MCTS controller
- new: **ended-slot-only MCTS ablation**

## What this comparison can and cannot claim

이 비교만으로 `ours가 MCTS보다 이론적으로 우월하다`고 말할 수는 없다.

현재 비교에서 방어 가능한 것은 아래뿐이다.

- 같은 tree / query-state / EMR / retrieval-pool semantics 아래에서
- ended-slot replacement 문제를
    - direct reseat로 푸는 것이 더 나은가
    - 아니면 local subtree planning으로 푸는 것이 더 나은가

즉 이 문서는 **controller-level empirical comparison memo**다.
이론적으로 더 좋다는 claim은, 추가 가정이나 더 강한 분석 없이 여기서 바로 나오지 않는다.

## Core algorithm

1. `run_round6.py`와 동일하게 rewrite prep을 하고 `query_state_after`, `query_post`를 만든다.
2. `frontier descendants + cumulative reached leaves`로 retrieval pool을 만든다.
3. active-slot continuation은 기존 round6와 동일하게 처리한다.
4. `ended_beam_count > 0`이면 ended-slot replacement 후보를 기존 round6와 동일하게 만든다.
    - `leftover_expandable`
    - 또는 `goexplore_direct_child`
5. ended slot은 **sequential fill**로 하나씩 채운다.
6. 각 slot마다 fresh MCTS search tree를 만든다.
    - stats는 그 reseat event 안에서만 유지하고, 다음 iter로 넘기지 않는다.
7. root child는 현재 slot에서 가능한 replacement branch들이다.
8. simulation은 현재 iter의 `selector_query = query_post`를 고정한 채 진행한다.
    - simulation 중 query rewrite는 다시 하지 않는다.
9. root child 하나를 고른 뒤, 그 branch subtree 안에서만 내려간다.
    - `direct_children`면 direct child만
    - `descendant_flat`면 strict descendant까지
10. 각 simulated node는 round6와 동일한 retrieval-scored branch rows를 사용한다.
11. reward는 해당 row의 branch score다.
    - 예: `meanscore_global`이면 `mean_score`
12. simulation 후 UCT backup을 하고, root child 중 visit가 가장 큰 branch를 그 slot replacement로 고른다.

### UCT rule in the current code

현재 `run_mcts.py`의 selection 값은 아래 형태다.

```text
q(child) + c * sqrt(log(parent_visits + 1) / (child_visits + 1))
```

- `q(child) = value_sum / visits`
- `c = MCTS_EXPLORATION_C`
- 방문하지 않은 child가 있으면 그쪽을 먼저 열어 보고, tie는 `immediate_reward`로 푼다.

즉 이 baseline은

- exploit: 지금까지 backup된 평균 reward `q`
- explore: 덜 본 child bonus

를 같이 써서 ended-slot replacement를 고른다.

### Parameter note

- `MCTS_EXPLORATION_C=1.4`
    - 완전히 임의의 숫자는 아니다.
    - 고전 UCB1/UCT에서 자주 쓰이는 `sqrt(2) ≈ 1.414` 계열의 default와 가깝다.
    - 다만 문헌에서도 exploration constant는 domain-dependent라서 고정된 정답값은 아니다.
- `MCTS_NUM_SIMULATIONS=32`
    - 이건 이론적인 표준값이라기보다 현재 task에서의 practical compute budget이다.
    - 큰 MCTS 시스템은 훨씬 더 큰 simulation budget을 쓰기도 한다.
    - 예를 들어 AlphaGo Zero는 move당 `1,600` simulations를 사용했다.

## What “synthetic root” means

여기서 `synthetic root`는 semantic-tree의 실제 root가 아니다.
그 iteration의 **replacement decision problem을 잠깐 감싸는 가짜 root**다.

예시:

- current selected branches: `(A,2)`, `(C,1)`
- 여기서 `(C,1)`이 ended됨
- 이번 iteration의 ended replacement 후보가 아래라고 하자
    - `(B)`
    - `(B,2)`
    - `(D,1)`

그러면 이번 slot의 MCTS tree는 이렇게 본다.

- synthetic root `R*`
- `R*`의 child = `(B)`, `(B,2)`, `(D,1)`

즉 `R*`는 “이번에 어느 replacement를 고를까?”라는 decision node일 뿐이다.
semantic-tree root `()`로 다시 planning하는 게 아니다.

### Tiny example

```text
selected before:
    (A,2), (C,1)

ended:
    (C,1)

replacement candidates:
    (B), (B,2), (D,1)

synthetic reseat tree:
    R*
    ├─ (B)
    ├─ (B,2)
    └─ (D,1)
```

이때 root-level retrieval pool은 아래다.

- branch component:
    - descendants of `(B)`, `(B,2)`, `(D,1)`
- plus:
    - cumulative reached leaves

즉 첫 root decision은

- “세 replacement 후보 중 무엇이 더 promising한가?”

를 보는 단계다.
- rewrite context = 현재 pre-hit top-k 문서 설명
- 최종 query는 항상 original_query + current query_state

### branch를 타고 내려가면 어떻게 되나

예를 들어 simulation이 root child로 `(B,2)`를 골랐다고 하자.

그 다음은 `(B,2)` subtree 안에서만 내려간다.

- `direct_children`이면 `(B,2)`의 direct child branch만 다음 후보가 된다
    - 예: `(B,2,p)`, `(B,2,q)`
- `descendant_flat`이면 `(B,2)` 아래 strict descendant branch 전체가 후보가 된다

그리고 이 child 후보들은 **현재 `query_post`를 고정한 상태에서** round6와 같은 retrieval-scored row로 점수화한다.

- child가 있으면 그중 하나를 UCT로 더 타고 내려간다
- child가 없으면 simulation은 그 node에서 끝난다
- 끝난 node의 reward를 backup한다

즉 `(B,2)`를 root child로 고른 뒤 `(B,2,p)`로 내려간다면,
그건 “replacement 후보 `(B,2)`를 선택했을 때 그 subtree 안의 continuation value가 어떤가”를 보는 것이다.

작은 subtree 예시는 이렇게 보면 된다.

```text
R*
└─ (B,2)
   ├─ (B,2,p)
   └─ (B,2,q)
```

이 단계의 subtree retrieval pool은 아래다.

- branch component:
    - descendants of `(B,2)` only
- plus:
    - cumulative reached leaves

즉 root에서는 `[(B), (B,2), (D,1)]` 전체를 비교하고,
`(B,2)`로 내려간 뒤에는 `(B,2)` subtree 내부 continuation만 본다.

### backup 이후

예를 들어 첫 3번 simulation이 아래처럼 끝났다고 하자.

```text
sim 1: R* -> (B,2) -> reward 0.62
sim 2: R* -> (D,1) -> reward 0.55
sim 3: R* -> (B,2) -> (B,2,p) -> reward 0.71
```

그러면 backup 후 root child 통계는 대충 이렇게 된다.

```text
(B):   visits=0
(B,2): visits=2, value_sum=1.33, q=0.665
(D,1): visits=1, value_sum=0.55, q=0.55
```

그 다음 simulation은 다시 `R*`에서 시작한다.
즉 backup은 끝이 아니라,

- 다음 simulation의 UCT 선택을 바꾸고
- 마지막에 root child 중 어느 branch를 실제 replacement로 쓸지 결정하는 통계를 누적하는 단계다.

모든 simulation이 끝나면 root child 중 visit가 가장 큰 branch가
이번 ended slot의 실제 replacement가 된다.

핵심은:

- root child = replacement 후보
- 그 아래 subtree = 그 replacement를 골랐을 때의 local continuation structure

## Locked decisions

- headline arm은 `leftover_expandable`
- `goexplore_direct_child`도 지원하지만 secondary arm
- `frontiercum_qstate` 유지
- EMR 유지
- `return + cumulative pool` 유지
- MCTS는 ended-slot replacement에만 적용
- ended slot이 여러 개면 sequential fill
- simulation 중 query는 고정
- stats는 **per reseat event**로 리셋
- reward는 기존 branch score 그대로 사용
- canonical full MCTS가 아니라 **task-specific ended-slot MCTS ablation**

## Claim boundary

방어 가능한 문장:

- `Under matched frontiercum_qstate retrieval, EMR, and query-memory semantics, the go-explore-inspired recoverable frontier controller outperforms an MCTS-style ended-slot controller on this retrieval task.`

## Bottom line

비교축은 아래로 고정한다.

- `ours`
    - `frontiercum_qstate + ended_reseat + leftover_expandable`
- `MCTS ablation`
    - same query-state
    - same EMR
    - same retrieval pool
    - only ended-slot replacement is switched to an MCTS-style controller

이게 지금 목적에 맞는 공정한 비교다.

## Is our go-explore inspired method better than MCTS?

- rollout에서 (B,2,p)가 좋아 보여도 자동으로 root action으로 올리면 안 됩니다.
  핵심은 “지금 할 수 있는 action”과 “rollout 중 미래에 도달한 state”를 구분하는 겁니다.
