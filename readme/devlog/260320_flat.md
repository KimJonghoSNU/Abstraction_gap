# 2026-03-20 Round6 flat-descendant ended-reseat

## 목적

`round6`의 현재 ended-reseat는 active beam 쪽에서 `direct child`만 다음 branch 후보로 본다. 이번 구현의 목적은 이 local continuation을 `descendant` 단위까지 넓혀서,

- `run_round3`의 flat branch retrieval 감각을
- `round6`의 tree-constrained retrieval pool 위에 얹는 것

이다.

핵심 질문은 두 개다.

- active branch를 `child-only`가 아니라 `descendant-flat`으로 고르면 더 좋은 중간 branch state를 찾는가?
- leaf를 만나 ended slot이 생겼을 때, replacement candidate를 `whole-tree flat`으로 둘지 `go-explore direct child`로 둘지 어떤 쪽이 더 나은가?

## 구현

### 1. Active-slot controller

새 option:

- `--round6_expandable_candidate_mode {direct_children, descendant_flat}`

이번 실험에서 기본은 `descendant_flat`이다.

의미:

- iter 0:
    - 전체 non-root non-leaf branch를 candidate로 둔다.
- iter 1 이후:
    - `selected_before`의 `strict descendant non-leaf branch` 전체를 candidate로 둔다.
- candidate scoring:
    - 현재 `round5_selector_mode`를 그대로 재사용한다.
    - 기본 실험값은 `meanscore_global`이다.
- selection:
    - slot quota 없이 `global union top-B`

즉, active slot은 더 이상 `direct child only`가 아니라, 현재 선택된 branch subtree 안의 더 깊은 branch로 바로 jump할 수 있다.

### 2. Ended-slot replacement

새 option:

- `--round6_expandable_ended_scope {leftover_expandable, whole_tree_flat, goexplore_direct_child}`

이번 실험에서 보는 두 새 scope:

- `whole_tree_flat`
    - semantic tree 전체 non-leaf branch를 candidate로 둔다.
    - flat retrieval 성향이 가장 강하다.
- `goexplore_direct_child`
    - root와 현재 선택된 branch들의 decision point에서, 아직 exploit되지 않은 `direct child`만 candidate로 둔다.
    - Go-Explore 해석에 더 가깝다.

기존 `leftover_expandable`은 baseline 호환용으로 남겨뒀다.

### 2.5 `leftover_expandable` vs `goexplore_direct_child`

이 둘은 비슷해 보이지만, 실제 candidate set의 의미가 다르다.

- `leftover_expandable`
    - 현재 `prediction_tree` 안에 이미 materialize되어 있는 expandable endpoint 중
    - 지금 beam에 안 들어간 것들을 ended-slot replacement 후보로 둔다.
    - 즉, 예전에 살아남아 있던 live frontier라면 더 깊은 branch라도 바로 복귀할 수 있다.
- `goexplore_direct_child`
    - root와 현재 선택된 branch들의 decision point에서
    - 아직 exploit하지 않은 `direct child`만 ended-slot replacement 후보로 둔다.
    - 즉, 현재/과거 경로에서 놓친 sibling branch를 다시 여는 방식이다.

예시:

```text
()
├─ (A)
│  ├─ (A,1)
│  └─ (A,2)
│     ├─ (A,2,x)
│     └─ (A,2,y)
├─ (B)
│  ├─ (B,1)
│  └─ (B,2)
│     └─ (B,2,p)
└─ (C)
   ├─ (C,1)
   └─ (C,2)
```

현재 selected branch가 아래라고 하자.

```text
selected_before = [(A,2), (C,1)]
```

그리고 현재 `prediction_tree` 안에 이미 살아 있는 expandable endpoint가 아래라고 하자.

```text
(A,2), (B,2), (B,2,p), (C,1)
```

이때 `(C,1)`이 ended 되어서 reseat 하나가 필요하면:

- `leftover_expandable`
    - 후보 = `(B,2)`, `(B,2,p)`
    - 즉 깊은 옛 frontier `(B,2,p)`로도 바로 돌아갈 수 있다.
- `goexplore_direct_child`
    - 후보 = `(B)`, `(A,1)`, `(A,2,x)`, `(A,2,y)`, `(C,2)`
    - 즉 현재/과거 경로에서 놓친 direct child만 다시 연다.
    - `(B,2,p)`처럼 깊은 branch로는 바로 teleport하지 않는다.

retrieval pool 쪽 차이도 여기서 나온다.

- `leftover_expandable`
    - candidate branch descendants에 이미 깊은 frontier subtree가 바로 들어갈 수 있다.
- `goexplore_direct_child`
    - candidate branch descendants는 direct-child subtree 단위라서 더 얕고 넓다.

즉 해석은 아래처럼 잡는 것이 맞다.

- `leftover_expandable`
    - wider live-frontier reuse
- `goexplore_direct_child`
    - more structured missed-sibling reopen

### 3. Materialize

descendant-flat / whole-tree-flat 쪽은 retrieval이 아직 prediction tree에 instantiate되지 않은 deeper branch를 바로 고를 수 있다.

그래서 선택된 branch path를 다음 iter traversal이 실제 beam state로 쓸 수 있게, `seed_beam_from_gate_paths(..., reset_history=False)`로 prediction tree 안에 materialize하도록 바꿨다.

의미:

- retrieval이 고른 deeper branch path를
- 다음 iter child-slate 생성이 가능한 실제 `beam_state_paths`로 바꾼다.

### 4. 유지한 것

이번 변경에서 일부러 안 바꾼 것:

- rewrite prompt
- rewrite evidence construction
- `query_pre`, `query_post`
- eval retrieval
- leaf를 만났을 때 ended slot만 replacement한다는 큰 semantics

즉 이번 실험은 rewrite ablation이 아니라 controller / branch-transition ablation이다.

### Branch-level findings

가장 중요한 초기 차이:

- iter 0 branch hit rate
    - `ours = 1.00`
    - `flat = 0.73`
- iter 1 branch hit rate
    - `ours = 0.95`
    - `flat = 0.52`
- iter 2 branch hit rate
    - `ours = 0.83`
    - `flat = 0.71`

즉 `flat`은 첫 2-3 step에서 **정답 branch region coverage**를 더 자주 놓친다.

동시에 depth는 초반부터 훨씬 깊다:

- iter 0 selected depth mean
    - `ours = 1.00`
    - `flat = 3.33`
- iter 1 selected depth mean
    - `ours = 1.94`
    - `flat = 3.36`
- iter 2 selected depth mean
    - `ours = 2.99`
    - `flat = 3.44`

즉 `flat`은 child-level coarse filtering 없이 곧바로 deeper descendant를 고르는데,
이게 일부 query에서는 좋은 branch를 더 빨리 찾기보다 **초기 top-level coverage loss**로 이어진다.

query-level로 보면 더 명확하다.

- 총 비교 query 수: `1384`
- gold-ancestor branch를 iter 0에서 맞춘 비율
    - `ours = 99.93%`
    - `flat = 73.48%`
- trajectory 전체에서 gold-ancestor branch를 한 번이라도 맞춘 비율
    - `ours = 100.00%`
    - `flat = 87.79%`

즉 `flat`은 약 `12.2%` query에서 끝까지 gold branch region에 들어가지 못한다.

이게 실제 end gap과도 연결된다.

- `flat`이 끝까지 gold branch를 한 번도 못 맞춘 query: `169`
    - 이 query들에서 `ours - flat` end nDCG 차이 평균은 `+1.40`
- 둘 다 iter 0에서 gold branch를 맞춘 query: `1017`
    - 이 query들에서 `ours - flat` end nDCG 차이 평균은 `-0.06`

즉 핵심은 이거다.

- `flat`이 항상 더 나쁜 게 아니다.
- **둘 다 초기에 올바른 branch region에 들어간 query에서는 둘 성능이 거의 같다.**
- 전체 평균 차이는 주로 `flat`이 초반 branch coverage를 놓치는 query들 때문에 생긴다.

### What this means

초기 가설은 `flat`이 deeper but noisier라서 나쁘다는 것이었다.
이건 절반만 맞다.

더 정확한 해석은:

- `flat`은 descendant-flat competition 때문에 초반 branch selection recall이 떨어진다.
- 즉 child-only expansion이 주던 **coarse-to-fine region coverage** 이득을 잃는다.
- 이후 iteration에서는 `flat`의 branch hit rate가 오히려 높아지는 구간도 있지만,
  그건 이미 놓친 초기 coverage를 완전히 만회하지 못한다.

즉 현재 `whole_tree_flat`의 약점은 instability보다 먼저,
**초기 branch-region recall 저하**라고 보는 게 맞다.

branch-state와 별개로, 현재 retrieval pool 자체가 gold doc을 아직 recover 가능한지도 봤다.

기준:

- `pool_has_any_gold_doc`

- **올바른 branch region을 depth 3~4까지 따라간 뒤, 그 지점에서 controller가 이탈하는 현상**에 가깝다.
- 그리고 iter `5+` recovery는, branch-state가 완전히 복구된다기보다
  **누적 leaf pool이 gold doc을 다시 보유하게 되면서 retrieval 쪽에서 만회하는 현상**에 더 가깝다.

즉 subset마다 정도 차이는 있지만, 전반적으로 **iter 3~4 exit** 패턴이 공통적이다.

### What this means

이 결과를 기준으로 보면, `ours`를 더 개선하려면 초반 coverage를 더 올리는 쪽보다

- depth `3~4`에서의 branch-state retention
- near-full reseat 직후 transition guard
- reseated branch / current gold-aligned branch를 다음 evidence에 연결하는 방식

이 더 중요하다.

## Ours dynamics: recovery, reseat depth, and per-dataset curves

1. 왜 `nDCG`가 iter `4 -> 5 -> 6`에서 다시 오르는가?
2. iter `3` 이후 reseat가 실제로 어느 depth에서 일어나며, 그 시기 middle-iteration drop은 dataset별로 얼마나 다른가?
3. 이 두 신호를 같이 보면 `ours`의 병목을 어떻게 해석해야 하는가?

### 1. Why does nDCG recover after the iter-4 dip?

`results/BRIGHT/analysis/round6_ours_dynamics_overall_iter.csv` 기준:

- `nDCG`
    - iter 4: `31.62`
    - iter 5: `35.97`
    - iter 6: `36.19`
- `branch_hit`
    - iter 4: `0.175`
    - iter 5: `0.120`
    - iter 6: `0.088`
- `reenter`
    - iter 4: `0.082`
    - iter 5: `0.059`
    - iter 6: `0.049`

즉 recovery 구간에서:

- `nDCG`는 오르는데
- `gold region branch hit`은 계속 내려가고
- `gold region re-entry`도 줄어든다

이건 중요한 point다.

- iter `4 -> 5 -> 6` 회복은 **정답 region branch가 다시 많이 잡혀서** 생긴 회복으로 보기 어렵다.
- 더 가능성 높은 해석은:
    - cumulative reached leaf memory
    - rewrite-retrieve loop
    - doc-level reranking
  이 branch-state와 독립적으로 ranking quality를 끌어올린다는 것이다.

즉 현재 `ours`의 later recovery는 branch controller recovery보다 retrieval/evidence-side recovery에 가깝다.

- 그리고 iter `5+`의 recovery는, branch hit / re-entry가 다시 강하게 오르기보다 retrieval/evidence-side에서 보정되는 성격이 더 강하다.

### 3. What this means

현재까지의 신호를 같이 보면:

- `flat`의 약점은 초기 branch-region recall 저하
- `ours`의 약점은 depth `3~4`에서의 branch-state retention 실패
- `ours`의 later recovery는 branch re-entry보다 retrieval/evidence-side 보정

즉 `ours`를 더 좋게 만들려면 다음이 맞다.

- branch-state를 더 오래 유지하게 하거나
- reseat된 branch를 다음 evidence state에 더 강하게 연결하거나
- 아니면 branch controller와 retrieval controller를 분리해, branch-state failure가 doc ranking까지 바로 무너지지 않게 해야 한다
