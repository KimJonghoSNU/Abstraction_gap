 # Round5 재구성 계획: agent_executor_v1 + retriever-MRR branch selector (MRR 전용)

  ## 요약

  - 목표: src/run_round5.py를 최소 실험 파이프라인으로 재작성한다.
  - 핵심 변경: 다음 branch 선택을 LLM traversal(prompt/slate) 대신 retriever 기반 Highest-MRR sub-branch로 대체한다.
  - 이번 범위: MRR selector만 구현 (DPO 비교 로직은 이번 구현에서 제외).
  - 참고 정렬: 현재 run_round5.py는 src/prompts.py 기반 traversal이 아니라 round4 category-policy 코드가 중심이다. 따라서 파일을 slim하
    게 재구성하는 방향이 맞다.

  ———

  ## 구현 목표(결정 확정)

  1. 엔트리포인트: src/run_round5.py 단일 파일에서 동작
  2. rewrite: 매 iter agent_executor_v1 강제 사용
  3. branch 선택: selected branch들의 자식 후보를 전부 모아 global top-B 선택
  4. MRR 정의: 각 후보 sub-branch의 best reciprocal rank = 1/(rank+1) (rank는 local retrieval 순위)
  5. MRR pool: top 100 leaf
  6. iter 0 규칙: retrieve -> rewrite를 먼저 수행한 뒤 root child에서 MRR로 beam 초기화
  7. 정리 범위: round4 category-policy/rule/fusion 관련 코드 제거(최소 파이프라인)

  ———

  ## 알고리즘 상세(결정 완료)

  각 sample별 iter 처리:

  1. 현재 branch 상태에서 local retrieval (pre-rewrite)
      - iter0에서 selected_branches가 비어 있으면 root 전체 하위 leaf를 pool로 사용
      - leaf pool에서 query(last_query 없으면 original query)로 top100 retrieval
      - 이 결과를 rewrite context(leaf_descs)로 사용
  2. rewrite (agent_executor_v1)
      - prompt 입력: original_query, previous_rewrite, leaf_descs, branch_descs(비워도 됨)
      - 출력 JSON의 Possible_Answer_Docs를 파싱하고 concat하여 new_query 생성
      - 캐시 hit/miss 모두 동일한 state 포맷으로 기록
  3. local retrieval (post-rewrite, branch selection용)
      - 동일 leaf pool에서 new_query로 top100 retrieval 수행
      - 이 랭킹을 기준으로 child 후보 branch MRR 계산
  4. 다음 branch 선택 (global top-B)
      - 후보 집합: selected_branches의 direct children 전체
      - 후보 branch 점수: best reciprocal rank
      - 전 후보를 합쳐 정렬 후 B=max_beam_size개 선택
      - iter0인 경우 root child 후보에서 동일 방식으로 초기 beam 생성
  5. 평가/로그
      - main ranking은 post-rewrite local retrieval 결과 leaf->doc_id 순서
      - nDCG@10, Recall@10/100/all, Coverage
      - iter_records에 아래를 저장:
          - selected_branches_before/after
          - candidate_branch_scores (MRR + tie-break score)
          - query_pre, query_post, rewrite_docs
          - local_paths/doc_ids, metrics

## 2026-03-03 업데이트: Legacy 알고리즘(현재 실행 기준)

현재 `results/BRIGHT/ndcg_summaryround5.csv`는 **legacy round5** (`agent_executor_v1 + retriever-MRR branch selector`) 결과로 해석한다.

### Iteration t에서의 실제 처리 순서

1. `query_pre` 결정
    - `sample.query`가 있으면 사용, 없으면 `original_query`.

2. pre-rewrite retrieval (context 구성)
    - 검색 대상: 현재 selected branch 하위 leaf pool (없으면 전체 leaf).
    - `query_pre`로 top-`round5_mrr_pool_k` retrieval.
    - 상위 문서 설명(`leaf_descs`)을 rewrite 입력 컨텍스트로 사용.

3. rewrite (`agent_executor_v1`)
    - 입력: `original_query`, `previous_rewrite`, `leaf_descs`.
    - 출력: `Possible_Answer_Docs`를 concat해서 `query_post = original_query + rewrite` 생성.

4. post-rewrite retrieval (평가 + selector 입력)
    - `query_post`로 같은 leaf pool에서 retrieval 수행.
    - 이 결과로 nDCG/Recall/Coverage 계산.

5. branch 선택 (MRR selector)
    - 후보: 현재 selected branch들의 direct child (iter0은 root child).
    - 점수: post-rewrite retrieval에서 후보 branch 하위 leaf의 best reciprocal rank.
    - global top-`beam_size` 선택.

6. 상태 업데이트
    - `selected_branches_after`, retrieval 결과, metrics를 `iter_records`에 저장.

핵심: branch 확장은 LLM traversal이 아니라 **retriever 점수(MRR)**로 제어하고, LLM은 query rewrite에만 사용한다.
